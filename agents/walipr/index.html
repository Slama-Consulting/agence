WaliPR

Inspecteur de pull requests

« Je relis chaque diff comme un dossier sensible. Mes remarques se posent là où elles font mouche : sur la ligne. »

Compatible : Claude Code Copilot Cursor

Mission

WaliPR relit vos pull requests et merge requests sur GitLab, GitHub et Bitbucket (Cloud + Server). Il analyse le diff Kotlin / Python / TypeScript / Java, traque les régressions, les fuites de ressources et les anti-patterns, puis pose ses remarques en commentaires inline, ligne par ligne, avec des patches prêts à appliquer quand c'est pertinent.

Capacités

Depuis la v1.10, WaliPR expose 9 outils MCP répartis en deux modes : un workflow agentique (recommandé) où l'agent IDE compose plusieurs outils pour aller chercher le contexte hors-diff, et un mode one-shot hérité (rapide mais superficiel).

Mode agentique (recommandé)

  • walipr_review_mr_orchestrate Point d'entrée. Renvoie un plan textuel FR/EN qui guide l'agent IDE étape par étape, sans appel LLM ni réseau.
  • walipr_get_mr_metadata Titre, description, branches, SHA — pour comprendre l'intention de la MR.
  • walipr_get_mr_diff Diff unifié + résumé par fichier touché.
  • walipr_get_mr_file Contenu complet d'un fichier à la SHA de la MR — pour donner du contexte autour des hunks.
  • walipr_submit_mr_review Poste les findings comme drafts (« Add to review ») par défaut. Pas de notification au reviewé tant que tu n'as pas publié.
  • walipr_publish_mr_review Publie tous les drafts en une review groupée (1 notification au lieu de N). Équivalent du bouton « Submit review » côté GitLab/GitHub.

Mode legacy (one-shot et local)

  • review_local_diff Review d'un diff git local (working tree, branche, range).
  • review_staged Review des changements stagés avant git commit.
  • review_merge_request One-shot. Voit uniquement le diff hors-contexte — réservé aux cas où la rapidité prime sur la profondeur.

Installer WaliPR

WaliPR est livré avec BugHound. Une seule commande installe l'agent et l'enregistre comme serveur MCP walipr dans votre IDE. 

curl -fsSL https://slama-consulting.github.io/agence/install.sh \
  | BUGHOUND_AGENTS=walipr bash

BUGHOUND_AGENTS=walipr indique à install.sh de n'enregistrer que WaliPR comme serveur MCP, sans toucher à SherleKhomes ni aux autres agents que vous auriez déjà installés.

Pour installer WaliPR et SherleKhomes en même temps : curl … | BUGHOUND_AGENTS=sherlekhomes,walipr bash. Si BugHound est déjà installé et que vous voulez juste enregistrer WaliPR : bughound mcp-install --agents walipr.

Configurer votre forge

Pour que WaliPR puisse lire et commenter une PR/MR, il a besoin d'un token de la forge concernée. Les tokens vivent dans ~/.bughound/.env, sur votre machine — jamais dans un dépôt git.

① Génère ton PAT sur ta forge

Page directe (gitlab.com) avec scopes pré-cochés : Créer un PAT GitLab →

Scopes : read_repository (lecture du diff) + api (post-comment inline).

Variable : WALIPR_GITLAB_TOKEN

GitLab self-managed ? Mêmes scopes, page /-/user_settings/personal_access_tokens de votre instance.

Page directe (PAT classique) : Créer un PAT GitHub →

Scopes : repo (privé) ou public_repo (uniquement public). repo couvre lecture et post-comment.

Variable : WALIPR_GITHUB_TOKEN

GitHub Enterprise ? Mêmes scopes, page /settings/tokens de votre instance.

Page directe (App Passwords) : Créer un App Password Bitbucket Cloud →

Permissions à cocher : Pull requests : Read + Pull requests : Write.

Variable : WALIPR_BITBUCKET_TOKEN

Sur Bitbucket Server / Data Center : ouvrez votre profil → Manage account → HTTP access tokens → Create token.

Scopes : Pull request : Read + Pull request : Write.

Variable : WALIPR_BITBUCKET_TOKEN (la même que Bitbucket Cloud — WaliPR détecte automatiquement le type d'instance).

② Lance le setup interactif

bughound setup --with-walipr

Saisie masquée. Le token est écrit directement dans ~/.bughound/.env (chmod 600) sans transiter par l'historique shell ni par le presse-papier.

③ Lance ta première review

# Lecture seule
bughound review-mr https://gitlab.com/group/repo/-/merge_requests/42

# Avec commentaires inline ligne par ligne
bughound review-mr https://gitlab.com/.../merge_requests/42 --post-inline

Premiers pas

  1. Configurez vos tokens : bughound setup --with-walipr (saisie masquée, par forge).
  2. Mode agentique (recommandé) — dans Cursor / Claude Desktop / Copilot, dites simplement : review <url-de-la-MR>. WaliPR enchaîne automatiquement metadata → diff → fichiers complets → drafts groupés.
  3. Mode CLI one-shot (rapide, hors agent) : bughound review-mr https://github.com/org/repo/pull/123
  4. Pre-commit local avant push : bughound review-staged
  5. Côté forge : les findings agentiques arrivent en drafts (« Pending review » sur GitLab, « Pending Review » sur GitHub). Vous pouvez relire / éditer / supprimer chaque draft avant de cliquer « Submit review », ou laisser l'agent appeler walipr_publish_mr_review pour publier d'un coup.

À noter : WaliPR ne modifie jamais le code du repo qu'il review. Il ne reviewe QUE les changements introduits par la MR/PR (jamais le code pré-existant non touché par le diff). Le contexte fichier complet sert uniquement à juger l'impact des hunks, pas à être commenté ligne par ligne.
